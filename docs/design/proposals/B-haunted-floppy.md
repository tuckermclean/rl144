# The Haunted Floppy

## 1. Thesis

The review I'm engineering for: *"I died on depth 3 to a goblin, and when I retried the same seed there was a faint, dimmer @ already standing where I'd died — it was me, an hour ago, walking the exact path I'd walked before. Then I found someone else's death note. Then I loaded a ghost file a stranger sent me and watched their whole failed run play out like a movie. None of this needed a server."*

Success is *legible otherness on a deterministic floor plan*. rl144 already has the one asset no other 190KB game has: `state = replay(seed, input_log)`, hash-proven identical across backends (`make xhash` already exists and already passes). That's not a feature to bolt Undertale/Peasant's-Quest cosmetics onto — it *is* the fusion, if we deliver it honestly. Undertale's "the game remembers you" becomes retry-same-seed plus a log the game can quote, word for word, because it's a replayable proof, not a flag. Peasant's Quest's fellow-bumbler comedy becomes ghosts: your own corpses, the resident sim-bot's corpse, and — the stated MMORPG win condition — other players' `.sav`-shaped files, exchanged with no server logic beyond byte relay. I am spending the content budget on making *other runs visible* rather than on more authored one-player spectacle, because the unreplicable asset here is the *proof*, and proofs only matter if someone else can see them.

## 2. The floppy at v1.0 — feature list + size ledger

Base game (batch-3 baseline): winnable 5-depth roguelike, torch clock, 3 monsters, 4 themes, 3 vaults, save/replay, daily/string seeds, world hash, two backends (minifb 192,144 B packed / term 169,724 B packed).

v1.0 additions and their packed-size cost, ranked by centrality to the thesis:

| System | KB packed (est.) | Why |
|---|---|---|
| Ghost substrate (GhostState, trim/replay, sim-bot-as-resident-ghost) | +10 | reuses `Game`/`save.rs` almost entirely — see §4 |
| Save v2 (retry byte, auto-ghost-on-death) | +2 | one new input byte, one version bump |
| Grounded death-message + emote const tables | +4 | short slot-filled ASCII, same machinery as `THEMES` |
| Ghost/corpse render overlay (dim/tint via existing `scale()`) | +3 | no new colors, no new glyphs |
| Spectator autoplay (`--watch <ghost>`) | +1 | `apply_input` driven from a file instead of a keypress |
| SVGA sprite/shape-mask pass (§3) | +45 | procedural, palette-indexed, no art files |
| Audio stingers (ghost-sighted / death / win / mercy, no music) | +14 | see §3 |
| Networking relay, `net` feature, off by default | +0 (default build) / +12 (net build) | std::net only, no crate |
| **Total over baseline** | **~+91 KB** | |

Projected minifb packed: ~283 KB (19.7% of 1,474,560 B budget, 80% margin). Term backend stays leaner (no sprite pass matters less over glyphs, no audio): ~200 KB. wasm/mobile is its own target with its own budget (§3). Comfortable margin for the whole roadmap in §5, not just v1.0.

## 3. Presentation architecture

**SVGA answer.** "SVGA style" at this scale means: don't ship pixel art, ship a *procedural shading pass* over the existing autotile precedent. Wall autotiling already picks box-drawing glyphs from neighbor bitmasks (`render.rs`); the SVGA pass generalizes that from glyph-selection to a small library of 16×16 1-bit shape masks (edge/corner/fill patterns, ~32–64 B raw each) shared across *all* four themes and re-tinted per theme via the existing `PAL_*` const table and `scale()` function — zero new crate, zero new color data. ~40 shape masks × ~64 B ≈ 2.5 KB raw, a few KB packed. Ghosts and corpses need *no new pixel data at all*: a ghost is the player's own glyph/sprite rendered through `scale()` at reduced alpha-equivalent brightness, exactly the mechanism already used for light-tier dimming. Target logical resolution stays the 80×30 cell grid (doctrine: the grid is engine API); the SVGA upgrade is what a cell rasterizes *to* (16×16+ shaded blocks instead of flat glyphs), a backend-local concern. `Cell{ch,fg,bg}` doesn't change.

**New core surface (justified under core/crust doctrine).** `render_cells` gains one parameter: `ghosts: &[GhostMarker]`, where `GhostMarker{x,y,depth,ch:u16,fg:u32}` is derived, per real turn, from each parked ghost `Game`'s own `px/py/depth`. This is the minimum surface growth that lets a backend draw ghosts *as ordinary cells* — no sprite-layer type, no z-order system, no new asset kind. It's justified because the alternative (backends peeking into ghost state directly) breaks the core/crust seam far more than one slice parameter does.

**Mobile — the most serious answer of the three, because ghosts need players and players are on phones.** No wasm-bindgen (it's a crate; dependency list is frozen). Instead: a `wasm32-unknown-unknown` target built from `std` alone, with `#[no_mangle] extern "C"` exports (`tick`, `input_byte`, `cell_ptr`) and a hand-rolled ~120-line JS glue file that reads the `Cell` buffer straight out of wasm linear memory into a `<canvas>` `ImageData`, and turns touch/swipe into the existing 0–5 input-byte vocabulary — same "any frontend that emits those bytes is a valid client" doctrine already governs minifb and term. This is *harder* than reaching for wasm-bindgen, and I'm choosing it anyway because it keeps the dependency list frozen and the wasm binary tiny (no bindgen glue bloat). Portrait vs. landscape is pure frontend recomposition, not a grid change: landscape keeps the existing horizontal layout; portrait stacks status/log below a top-anchored, width-scaled map — same "frontends scale the fixed buffer" rule that already gives minifb its resizable window. Ghost exchange on a phone (no filesystem drama): a ghost file is ~1.5–2 KB (see §4) — small enough to round-trip through a QR code (flagged as a stretch item, not a v1.0 promise: QR alphanumeric capacity tops out around 2.9 KB, tight against worst-case logs) or a share-sheet/clipboard blob. This is what makes "players are on phones" a design constraint rather than an afterthought: the whole ghost economy is worthless if it only runs on desktops with a filesystem.

**Audio — commit, but narrower than the old 20–30 KB roadmap line.** No tracker song, no music loop: music doesn't advance legibility of other runs, so I cut it. Instead: a tiny square-wave/noise synth (~6–8 KB of engine) driving a handful of const stinger tables (~6 KB) for exactly the events that make an *other* presence legible — ghost sighted, ghost death replayed, your own retry beginning, a mercy choice taken. ~14 KB total, defended by the same principle as the sprite cut: spend bytes on making other runs perceptible, not on private atmosphere.

## 4. The fusion design

**Ghost file format** (`RLG1`, sibling to `RL14`): magic(4) + version u8(1) + seed u64 LE(8) + world_hash u64 LE(8) + outcome u8 (died_combat/died_dark/won/abandoned) + final_depth u8 + turns u32 LE + label: u8-length-prefixed ASCII ≤16 B (chosen from a const preset phrase table, Souls-style — no free text, so zero localization/moderation surface) + the raw input log. Trim rule: truncate trailing no-op bytes past a terminal state (dead/won); cap at the solver's own worst-case round-trip budget (1503 turns, `START_LIGHT`'s derivation) — a log that long is already a pathological outlier, not a ghost worth carrying. Header ~30 B, log ≤~1.5 KB typically far less for an efficient player: a ghost is a couple of kilobytes. These are runtime user data, exactly like `.sav` files — not shipped assets, so "no asset files" is untouched.

**Playback: parallel `Game` instances, stepped in lockstep.** A `GhostState` is a parked `Game::new(ghost.seed)` plus the trimmed log and a cursor. On every real player turn, each active ghost advances exactly one of *its own* logged inputs via the same `apply_input` the player uses — zero new simulation code, the existing engine *is* the ghost engine. Ghosts only render meaningfully when `ghost.seed == player's live seed` (daily seed, retry-same-seed, or a shared challenge code) — cross-seed ghosts don't overlay coherently on a different dungeon, so they play back solo via `--watch` instead. I'm stating that limitation up front rather than papering over it (see §7).

**Corpses without a worldgen MAJOR.** A corpse is a ghost overlay frozen at its death tile, not an entry stamped into `g.items`/`g.map`/`LevelState`. It never touches `dump()`, `solve_seed()`, or the golden fixtures — a clean checkout with no ghost files present produces byte-identical goldens, so the base game's MAJOR-version doctrine is untouched. The one place ghosts touch grounded flavor: a corpse present on entry may trigger a slot-filled line from the existing `flavor_rng` channel template table ("someone else came through here") — honest under the grounding doctrine, because a corpse is something the engine can *prove* happened, unlike invented lore.

**Save v2 semantics — retry vs. reroll.** Input byte 5 stays RESTART (reroll: `h64(seed,["restart"])`, as today). New byte 6, RETRY: `Game::new(g.seed)` — same world, another attempt, and it *auto-saves the just-ended attempt's log as a ghost* before resetting, so every death produces a ghost with zero extra player action. This is a save-format MAJOR (the input-byte vocabulary grows, versioned `SAVE_VERSION=2`) but explicitly *not* a worldgen MAJOR: it only changes which seed a new run starts from, never a channel draw order, so existing golden fixtures are unaffected. Old v1 logs (which never contain byte 6) still replay identically under `replay()`.

**Endings.** No new win states — the existing win/lose pair stays — but the End screen gains a slot-filled clause driven by a boolean the engine already has the data for: `ghost_encountered` (set when the player walks adjacent to an active ghost marker this run). "You were not alone" is earned, not authored.

**Writing plan.** Everything player-facing is short, slot-filled, ASCII, and grounded — extending the existing `THEMES`/`VAULTS` register, not inventing a new one. Two new const tables: death-message templates (killer/depth/turns slots, ~15 templates × ~60 chars ≈ 1 KB) and a preset ghost-label phrase list (~24 phrases × ~16 B ≈ 400 B). No free-text input, ever — that's a deliberate cut (§6), not an oversight.

**What a "server" is.** A dumb byte relay behind a `net` feature flag, off by default (0 B cost to the default build). No crate: `std::net::TcpStream`, a length-prefixed frame of (client_id, turn_counter, input_byte), rebroadcast to peers. A connected peer is *architecturally identical to a ghost* — a parked `Game` fed a byte stream — except the stream arrives live over a socket instead of from a file. This unifies local ghost racing and live multiplayer under one `GhostState{game, source: File(Vec<u8>) | Socket(TcpStream)}` type; no new "other player" entity kind is needed anywhere in `game.rs`. World state is never sent, only bytes — matches the direction memo exactly. The relay server is infrastructure, not a shipped target, so it doesn't count against the 1.44MB budget; it's a separate, unbudgeted binary.

## 5. Roadmap (batches 4..N, each with its gate and size checkpoint)

- **B4 — ghost substrate.** File format, `GhostState`, trim rules, sim-bot log capture (the existing `--sim` bot's `apply_input` calls recorded, not just aggregated), `--watch <ghost>` spectator autoplay. *Gate:* new `make ghost` target — round-trip trim→replay→hash, assert trimmed tail `state_hash` matches full replay. Checkpoint: +10 KB.
- **B5 — save v2.** Retry byte, auto-ghost-on-death/retry/reroll, version bump. *Gate:* backward-compat test (v1 logs still replay byte-identical), new v2 replay test, `xhash` extended to a v2 sample. Checkpoint: +2 KB.
- **B6 — ghost rendering + corpses + grounded messages.** `GhostMarker`, `render_cells` surface growth, death-message/label const tables. *Gate:* new frame-golden fixture with a canned ghost present (deterministic); an explicit test asserting `dump()`/`solve_seed()` output is byte-identical with or without ghost files in cwd, protecting the "no MAJOR" claim in §4. Checkpoint: +7 KB.
- **B7 — audio stingers.** Synth + event table, no music. *Gate:* size checkpoint only; `--mute` compile/run smoke check. Checkpoint: +14 KB.
- **B8 — SVGA shape-mask pass.** Procedural 16×16 masks, palette retint, ghost/corpse dimming via existing `scale()`. *Gate:* frame goldens re-baselined (presentation-only per doctrine, not seed-breaking — say so in the commit), human playtest flagged. Checkpoint: +45 KB.
- **B9 — wasm/mobile backend.** No wasm-bindgen, hand-rolled JS glue, touch→byte mapping, portrait/landscape recomposition. *Gate:* new `make check-wasm` — builds the target and runs `--dump`/`--solve` through it if a runtime (e.g. `wasmtime`) is available in the sandbox; **flagged honestly** as needing interactive playtest for touch feel regardless. Own size budget tracked separately.
- **B10 — networking relay (`net` feature, off by default).** std::net lockstep, dumb relay binary (unbudgeted). *Gate:* in-process loopback test — two `GhostState`s exchanging bytes over a fake in-memory transport (no real socket needed in a sandboxed CI), asserting converged `state_hash`; `net`-enabled build gets its own size checkpoint. Checkpoint: +12 KB (net build only).
- **B11 (stretch, low confidence) — live daily-seed ghost racing, QR ghost exchange polish, cross-play cohort UI.** Explicitly not promised for v1.0.

## 6. Cuts

- **Free-text chat/messages.** Souls-style presets only. Free text is a moderation and localization problem this project has permanently ruled out; presets keep the whole feature at const-table cost.
- **Full inventory UI, ranged combat, monster-variety expansion.** Deferred past v1.0 on purpose — more single-player systems don't make ghosts more legible, they compete with the bytes that do.
- **Tracker-style music.** Cut outright, not deferred (see §3): the old 20-30KB roadmap line bought private atmosphere; stingers buy shared-presence legibility for a third the cost.
- **wasm-bindgen, any JS framework, any matchmaking/account service.** Dependency-frozen doctrine and "server is a relay, not a platform" both rule these out.
- **Cross-seed live ghost overlay.** Two players on different seeds don't get to see each other move in real time — only exchange non-interactive replays. I'm not pretending this is solved; it isn't, and forcing it would mean either serializing world state (banned) or generating a shared seed for every social pairing (a real feature, just not this one).

## Rebuttal round

**A's flaw.** A explicitly stacks a second, correlated balance gate on top of a system already flagged fragile, and says so itself: `sim-band.json`'s own comment shows win 14.6%, band [10,25]%, 100% of deaths combat, and A's own §7 admits "it's plausible both the greedy-bot and pacifist-bot gates destabilize together, and chasing two sim-bot bands into a defensible correlated state could eat more engineering time than the writing I'm trying to protect." That's not a strawman, it's A's own risk assessment — a proposal whose central mechanic (ACT/regard/violence-tax) is engineered to destabilize the one gate this codebase already treats as touchy is a real cost, paid before a line of dialogue ships.

**C's flaw.** C bets the entire cross-platform audio story on hand-rolled `libasound.so` FFI, Linux-only, and its own §7 calls this "unproven on Windows/macOS," adding "write your own FFI per platform is exactly the kind of bespoke maintenance tax a mature crate like `cpal` exists to absorb." C ships that admission alongside a wasm mobile section whose byte estimate is "explicitly a guess" (§2) and whose touch/audio path is "asserted with byte-math confidence and implementation uncertainty" (§7 item 3) — the two platforms this fusion most needs (a phone, a non-Linux desktop) are the two C is least sure about, by C's own words.

**What I'd steal, and how.**
- **From C: the `scene()`/`SceneEntity` core surface**, not my thinner `render_cells(ghosts: &[GhostMarker])` param. C's insight — a `Cell` carries no identity across frames, no facing, no "still alive next frame" continuity — is exactly the gap ghosts have too: a ghost overlay needs the same continuity a live sprite needs. I adopt `scene()` wholesale and route both live entities *and* ghost markers through one `SceneEntity` stream instead of a parallel ghost-only struct. One core surface, not two competing ones — C's own §7 flags that exact convergence risk if two proposals both want scene-level data; I'm resolving it in favor of the model that already unifies more.
- **From C: the palette-ramp + Bayer-dither retint trick** for ghost/corpse tinting, replacing my flatter `scale()`-only dimming — same zero-new-geometry principle, more visually distinct "this one's a ghost" read for near-zero extra bytes.
- **From A: the violence tax** (`VIOLENCE_TAX: i32 = 1` off `self.light` per attack, one line in `try_move_player`). I'm not adopting A's ACT/regard/spare system — that's real balance-gate risk I don't need — but the tax alone, with no mercy verb attached, is free and directly serves my thesis: a ghost's light-burn curve becomes a diagnostic a watching player can read ("this ghost fought constantly, that's why it died at depth 3 with light to spare") without authoring a single new line of text. It also answers the batch-3 combat-lethality finding I otherwise leave untouched.

**Revised final stance.** I concede A's sharpest point applies to me too, partly: ghosts alone, with zero authored world to anchor them, risk reading cold — a stranger's death is only affecting if the world around it has some voice, so I'm not cutting `TONE_LINES`-style grounded flavor, just not expanding it into dialogue trees. Adopting `scene()` from C strengthens my ghost-rendering story for free and resolves a core-surface collision before either proposal has to re-litigate it later. Adopting the violence tax from A costs one line and turns my main untouched finding (combat lethality) into ghost-legible data instead of leaving it as a known gap. The thesis is unchanged: replay-as-portable-proof is still the one asset no rival can replicate, and it's now built on a shared, better core surface plus one borrowed one-liner that makes every ghost a more honest witness.

The honest case against this proposal: **the MMORPG payoff is aspirational without population.** With zero other players, "the tiny MMORPG" degrades to your own corpses plus one resident sim-bot ghost — real Undertale-memory value, but a much smaller claim than "asynchronous multiplayer." The relay server that would make it real is explicitly out-of-scope infrastructure, not something this floppy ships or guarantees exists. Second, the same-seed-only overlay constraint (§4) narrows "MMORPG" to daily-seed/challenge-code cohorts, not an open shared world — a rival proposal could fairly call this "async leaderboard cosplay," not multiplayer. Third, the wasm-without-wasm-bindgen bet is real engineering risk: hand-marshaling touch, canvas, and (for B10) a browser transport by hand is more JS-glue debt than it sounds, and if it slips, the "mobile matters most to me" claim goes unproven. Fourth, N parked `Game` instances per popular ghost cohort is cheap per-ghost but has no designed cap/culling policy yet — a hot daily seed with hundreds of ghosts could get visually noisy or slow, and I haven't sized that. Fifth, and most fundamentally: this bets that *proof of another run* generates more delight per byte than *authored content* does. That's a judgment call about what makes a roguelike memorable, not something I can prove from the repo — a reviewer who wanted more hand-written comedy and story beats will find this proposal thin on exactly that.
