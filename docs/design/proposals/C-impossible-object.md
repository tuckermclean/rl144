# Proposal C — The Impossible Object

## 1. Thesis

Nobody screenshots a JSON balance table. The one-line review I'm engineering for: *"wait, this fits on a floppy?"* — said after the player has already fallen for the game, not before. rl144's systems (round-trip amulet, light-as-clock, three monster kinds) are sound but mute; a bump reads as a number changing, a death reads as a status line. Peasant's Quest and Undertale are both, at bottom, **presentation disciplines**: PQ's comedy lives in how absurdly a death is staged, Undertale's intimacy lives in a leitmotif that returns transposed once you've earned it. Neither needed more systems than we have. I'm proposing we spend the 1.28MB of headroom on making every existing verb feel authored — pixel art, a synth, screen-feel, real mobile — and treating that as the actual v1.0 deliverable, with game-system growth kept deliberately modest. If a rival's proposal outspends this one on new mechanics but the game still *looks* like an ASCII roguelike wearing a color palette, they've lost the review.

## 2. The floppy at v1.0 — feature list + size ledger

**Everything already shipped** (baseline, unchanged): worldgen, torch clock, 3 monsters/4 themes/3 vaults, save/replay, daily seed, sim-gated balance, minifb + term backends. Packed today: minifb 192,144 B, term 169,724 B (both measured, `make check` baseline).

**New in this proposal**, native (minifb) target, packed-byte estimates below. Methodology: raw byte math first, then a compression multiplier *inferred from this binary's own measured ratio* (506,304 B stripped → ~166,000 B packed today ≈ 3.05× whole-binary average). I split code and data because they don't compress alike: repetitive const tables (palette ramps, mostly-transparent sprite nibbles) compress well past that average; branchy logic compresses worse. I use 6× for sparse pixel data, 2.5× for logic, both conservative versus demoscene norms, and pad the code estimate ×2 as a pre-implementation safety fudge — AGENTS.md's own rule is "measure, don't guess," so every line below gets a `cargo bloat` checkpoint at its batch (see §5), not a promise.

| System | Raw | Packed est. | Note |
|---|---|---|---|
| Sprite base shapes (16×16/24×24, 4bpp nibble-packed, see §3) | 10.75 KB | ~2 KB | mostly zero/background nibbles + reused ramps |
| Compositor + outline + dither pass (code) | ~15 KB | ~6 KB | palette-ramp lookup, 1px silhouette outline, Bayer 4×4 dither |
| Animation (walk bob, hit squash, idle breathe — code, LUT-driven) | ~10 KB | ~4 KB | no extra sprite frames; math over the base shapes |
| Particle/screen-feel (spark, dust, palette flash — code) | ~8 KB | ~3 KB | deterministic, render-only (see §3) |
| Scene-layer core API (`scene()`, `SceneEntity`) | ~3 KB | ~1.5 KB | additive core fn, zero Cell/golden impact |
| Audio: synth core (<150 lines, osc/env/FM/noise/mixer) | ~6 KB | ~2.5 KB | zero platform calls, in core |
| Audio: ALSA playback shim (hand-rolled FFI, backend-local) | ~5 KB | ~2 KB | Linux only this pass; see §3 for the doctrine call |
| Audio: tracker pattern data, all themes + leitmotif table | 0.6 KB | ~0.2 KB | tiny integers, highly repetitive |
| **Native subtotal, padded ×1.5 for estimate risk** | | **~33 KB** | |

**Native v1.0 packed total: 192,144 + ~33,000 ≈ 225 KB (15.3% of 1,474,560 B), 1.25MB margin — an 84% safety margin, not the required 20%.** I'm not spending less than a rival by choice; the technique is just this cheap. §7 argues why banking that margin instead of spending it further is itself defensible.

**Mobile/wasm target (separate budget, "per target" per brief):** new `backend-wasm` feature, no new crate (font8x8 reused, minifb dropped for this target — no x11 dependency at all). Estimate 120–180 KB after `wasm-opt -Oz`; UPX doesn't run on wasm, so the wasm-analog "pack" step is `wasm-opt -Oz` + the server's gzip/brotli, and that combined number is what gets checked against 1,474,560 B. This is a genuine unknown until built — flagged, not asserted (see §5 batch 7, §7).

## 3. Presentation architecture

**SVGA pixel art with zero asset files.** Two options, both possible, chosen deliberately:
- *Raw 4bpp const tables*: 16×16 = 128 B/sprite, 24×24 = 288 B/tile. A full roster (player 12 walk frames, 3 monster kinds × 2 frames reskinned per theme via palette swap not new geometry, 4 item shapes, 16 wall-autotile shapes + 2 floor + 2 stair at 24×24, 6 environmental decorations, 4 particle shapes, 8 UI-chrome pieces) totals **10,496 B raw geometry** — genuinely tiny before compression even runs.
- *Pure runtime procedural generation* (SDF blobs, generated silhouettes): more code, not less — a shape generator is typically larger compiled than the 128-byte table it would replace, and code compresses *worse* than data here. Procedural generation only wins when you need combinatorial variety from a small ruleset; we don't — a rat looks like a rat once.

**Decision: hybrid, demoscene-standard.** Hand-authored base geometry (small, controllable, art-directed) + procedural *post-processing*: one palette ramp per theme (16 colors × 3B = 48B) recolors the same rat/goblin/ogre silhouette four ways; a Bayer 4×4 ordered-dither pass (16-byte const matrix) gives the "SVGA" grain the brief asks for without per-pixel authored dithering; a 1px silhouette-outline pass (Sobel-style, computed not stored) reads at both 16px and scaled-up sizes. This is the actual old-SVGA-game trick (Ultima/NetHack tile reskinning) — geometry once, variety from ramps.

**The scene layer — new core surface, justified.** `render_cells` throws away identity: a `Cell{ch,fg,bg}` for a monster glyph carries no facing, no animation phase, no "this is entity #7, still alive next frame" continuity — exactly what a sprite backend needs to not look like a slideshow. I'm proposing one additive core function, in `render.rs` (already the core module that walks `Game` into presentation data — same seam, richer output):

```rust
pub(crate) enum SpriteKind { Player, Monster(MKind), Item(IKind), Tile(TileShape) }
pub(crate) struct SceneEntity {
    pub(crate) kind: SpriteKind,
    pub(crate) x: i32, pub(crate) y: i32,      // same grid coords as Cell, no new coordinate system
    pub(crate) facing: Facing,                  // last nonzero move direction
    pub(crate) anim_phase: u8,                  // g.turns-derived, pure fn, no new state to persist
    pub(crate) light_pct: u32,                  // reuses render.rs's existing light_pct()
}
pub(crate) fn scene(g: &Game) -> Vec<SceneEntity>
```

Zero platform calls, zero cfg — stays core. It does **not** touch `Cell`, `render_cells`, or the frame goldens: term backend is untouched, `--dump`/`--render-frame` byte-identical, no MAJOR bump. The only new *state* is `facing` (derived from the existing dx/dy of the last `try_move_player` call, one field on `Monster`/`Game`, not persisted to save, not printed by `--dump`) — a genuinely additive, presentation-only field. This is the textbook core/crust case the doctrine asks a backend to make explicitly, and it's a narrow one: one function, one struct, no behavior change to anything headless verifies.

**Mobile: portrait/landscape.** No new core surface needed here — the viewport crop is backend-side windowing over the *existing* 80×30 Cell buffer, same trick either backend already does by scaling pixels. Landscape: full 80×30, letterboxed (current behavior, unchanged). Portrait: crop to 40 visible columns centered on the player (`clamp(px - 20, 0, 40)`), full 30 rows — a 4:3-ish visible slice that reads on a phone; status/log rows are the *same* `Cell` rows `MAP_H`/`MAP_H+1..`, just cropped and rendered at a larger per-cell pixel scale, not a redesigned UI. Touch: a fixed on-screen d-pad (4 quadrants) + a wait/interact tap zone, translated in the wasm host to the *same* input bytes 0–4 (5/restart reachable from a menu tap) — the frozen vocabulary is exactly what makes this cheap; no new input plumbing in core at all.

**Which backend ships mobile.** `minifb` doesn't do wasm/touch. Rather than add a crate (`web-sys`/`wasm-bindgen` are not "impossible to hand-roll in <150 lines" by a wide margin — they're whole binding generators), a new `backend-wasm` feature exports a handful of `extern "C"` functions (`rl144_frame() -> *const u8`, `rl144_input(u8)`) and imports `canvas_put_pixel`/`audio_push_samples` from a thin (~50-line) JS host. That JS host is the mobile equivalent of the OS hosting the native binary — no browser can run a bare wasm module without *some* host page, same as no OS runs a bare ELF without a loader — so it sits outside the byte ledger, same accounting line as "the window manager isn't counted for minifb."

**Audio: commit, don't cut.** The old roadmap reserved 20–30 KB; my design comes in under that (§2: ~4.7 KB packed for the whole system on the native target). Core synth (`src/audio.rs`, new core module, zero platform calls): square/triangle oscillators via a small LUT, one ADSR envelope, a 2-op FM voice (carrier + modulator, fixed modulation index per instrument), and a noise voice via a *dedicated* LFSR reseeded per-note from the note index (a pure function of (theme, note, tick) — deliberately **not** drawn from any `channel()` stream, so audio can never perturb worldgen/combat/ai, and never needs to per the "never let one channel leak" rule since it isn't a channel at all). A tracker-style sequencer steps const `Pattern` tables (theme ambient loop, combat sting, one shared leitmotif melody encoded as *relative pitch deltas* so it transposes per theme/register — the actual Undertale trick, and cheap: one 24-step delta table, not four full melodies). Target: under 150 lines for the synth core, hand-counted at design time, hard-capped at implementation.

Playback is the genuinely hard part, and it's a *platform* problem, not a synthesis problem — so it's a backend concern like `minifb`'s window loop, not a new dependency. Linux: hand-rolled `libasound.so` FFI (open PCM device, set hw params, `snd_pcm_writei` loop) — well under 150 lines, no crate. This is a real doctrine bet: `cpal` would derisk Windows/macOS in one dependency, at a size and "no engines"-adjacent cost I'd rather not pay before it's proven necessary. Windows/macOS playback backends (`waveOut`/CoreAudio FFI, same hand-rolled pattern) are scoped as their own small follow-on batches, not blocking v1.0's native (Linux-verified) target.

**Screen-feel**, all deterministic and render-only (no RNG channel touched, so no golden/solver/sim risk): hit squash (1-frame Y-scale on the struck sprite, computed from `g.turns` parity), palette flash (one frame of `scale()`-brightened tint on damage, reusing the existing dim/scale machinery in `render.rs`), particle glyphs (spark on hit, dust on move, motes drifting in low light — spawned from game *events* already logged, positioned/aged by a pure function of tick count, never stored in `Game`). None of this is new game state; it's derived display, which is exactly what keeps it out of the golden/save-format blast radius.

## 4. The fusion design

I'm not proposing new verbs. PQ's comedy and Undertale's intimacy are both *staging* of verbs we already have (move, wait, bump-attack, walk-over pickup). The addition is:

- **Every existing outcome gets an authored reaction.** A bump-into-wall (already free, already silent) gets a 1-frame squash + no sound (the joke is that nothing happens — pure PQ). A monster kill gets a kind-specific micro-animation + a `content.rs`-table death line (`DEATH_LINES[MKind]`, same authored-const pattern as `THEMES`/`TONE_LINES`, same grounding rule: restate what happened, never invent). A player death gets full-screen palette drain synced to the existing `killer` field — already computed, never staged.
- **Save v2, minimal: a memory ledger, not a mercy system.** One new field in the save trailer (versioned, `RL15` header) — a small per-monster-kind death counter (`[u16; 3]`), carried across restarts of the *same rerolled lineage* (the `h64(seed,["restart"])` chain already implies "this world remembers your resets"; this makes that literal). On re-encountering a monster kind you've died to before, one callback line fires (`content.rs` table, indexed by count tier) and its sprite gets a one-shade darker tint. That's the entire "the game knows you" mechanic — a data field and a palette lookup, not a spare/mercy combat system. Genuine mercy (sparing a monster) is explicitly cut (§6) — it's a systems addition, not a presentation one, and it's someone else's thesis to win or lose on.
- **Endings stay exactly two** (win/death), presentation-elevated: the End screen composites the run's dominant theme's palette + a static player+amulet silhouette (reusing sprite geometry, no new art) and closes with the leitmotif reprised in a resolved register on win, a minor/detuned register on death — the transposition table from §3 doing double duty.
- **Writing plan.** Registers: comedic for the free/low-stakes actions (bumps, waits, flavor pickups), sincere for lore/tier-3 inscriptions and the ending lines (already the pattern `content.rs` establishes with `THEMES`/`lore`). Volume: stays inside the existing discipline — log lines <78 chars, new tables sized like `TONE_LINES` (2 lines × N kinds), not sprawling. Authorship: I write the new `DEATH_LINES`/`MERCY_LINES`(memory-callback) tables myself, following the standing grounding rule, reviewed the same way `THEMES` was — const data, no new authoring pipeline.

## 5. Roadmap

Each batch: scope, verification-gate extension, size checkpoint. All native-target work; batch 7 is mobile.

1. **Batch 4 — scene layer + sprite foundation.** `scene()`/`SceneEntity` in core; base geometry tables, ramps, dither/outline compositor in `backend_minifb`. Gate: new unit tests on compositor purity (same input → same pixels); `--dump`/`--render-frame` goldens untouched (proves no core leak); `cargo bloat` checkpoint replaces the §2 estimate with a measured number. Term backend untouched.
2. **Batch 5 — animation + screen-feel.** Walk/hit/idle animation, particles, palette flash. Gate: new determinism unit tests (event+tick → identical particle set); size checkpoint. Flagged for human playtest (visual feel, unlike headless-verifiable determinism).
3. **Batch 6 — audio.** `src/audio.rs` core synth + ALSA backend shim + pattern tables. Gate: new `--dump-audio <seed>` headless mode prints the sequencer's note-event stream as text (extends the "any new system needs a headless observation point" rule exactly like `--dump`/`--sim` do) — a new golden-style text comparison joins `make check`. Size checkpoint.
4. **Batch 7 — mobile (wasm).** `backend-wasm` feature, JS host, touch mapper, portrait viewport crop. Gate: `make check` grows a third build leg (`--target wasm32-unknown-unknown`); touch→input-byte mapping gets a pure-function unit test; `wasm-opt -Oz`+gzip size checkpoint against 1,474,560 B, reported honestly even if it fails on the first attempt.
5. **Batch 8 — save v2 memory ledger + authored reactions.** `RL15` save format, `DEATH_LINES`/memory-callback tables. Gate: `--replay` on an `RL15` save must still hash-match; old `RL14` saves must still load (back-compat check added to `make check`).
6. **Batch 9 — SVGA richness pass** (optional, budget-gated): more monster/environment variants, per-theme music variation, End-screen illustration compositing. Only proceeds if batches 4–8 are under their estimates; explicit go/no-go against the size ledger, not assumed.

## 6. Cuts

- **No new verbs, no mercy/spare mechanic.** Presentation elevates the verbs we have; a real mercy system is a game-design bet this proposal doesn't make (it's the natural target for a rival's systems-first thesis — let them own it).
- **No sampled/recorded audio, ever.** Synth-only, by doctrine (no asset files) and by taste (a chiptune voice is more honestly "impossible object" than a compressed sample would be).
- **No parallax/scrolling backgrounds.** At 640×360 logical pixels the return is marginal and the byte/complexity cost isn't; the dither+ramp trick already does more for "SVGA-feel" per byte.
- **No native mobile wrappers (Swift/Kotlin/Capacitor).** wasm-in-browser is the only mobile path; app-store packaging is out of scope for a presentation thesis and would break the dependency doctrine outright.
- **No inventory UI.** Still someone else's signed-off batch; walk-over pickup is doctrine until then, and I have no presentation reason to force the sequencing.

## 7. Attack surface — the honest case against this

1. **Headless-first is this codebase's core discipline, and presentation is exactly what headless can't see.** Every animation/screen-feel batch above ends in "flagged for human playtest." A proposal that adds this much backend surface adds proportionally more code no CI gate actually exercises — the opposite of where this repo has invested its credibility so far.
2. **The hand-rolled-ALSA bet is real risk, not just cleverness.** It's cheap and dependency-clean on my Linux box; it is *unproven* on Windows/macOS, and "write your own FFI per platform" is exactly the kind of bespoke maintenance tax a mature crate like `cpal` exists to absorb. If it turns out brittle, the KB saved will look small next to the engineering hours spent.
3. **Mobile is the least de-risked third of this whole proposal.** The wasm audio/touch path is asserted with byte-math confidence and implementation uncertainty — "solve it properly" in §3 is a design, not a working prototype. The size estimate for the wasm target is explicitly a guess (§2).
4. **This proposal is a bet that presentation is now the higher-leverage spend than further balance work.** Batch 2's original finding (0% bot win rate, 100% combat deaths) was already fixed in batch 3 — the sim bot now wins 14.6%, gated in `tests/sim-band.json`'s [10,25]% band — so this is not a gamble against an open problem. The real risk is narrower: that band is new and tight, and every combat-adjacent change in §5 (kind-specific death animations, screen-feel timing, memory-callback tint shifts) is presentation-only and shouldn't move `--sim`'s numbers at all — but "shouldn't" isn't "verified," and I haven't specified a regression check that proves it. That's a gap worth closing in batch 4's gate, not a reason to distrust the ledger.
5. **The `scene()` core addition, however narrow, is still a precedent.** Once one backend gets richer-than-Cell data, the crust doctrine's "backend needing a new API must justify it" bar gets tested for real for the first time — if the other proposals also want scene-level data for their own reasons, we should be converging on one API, not two competing ones merged after the fact.

## Rebuttal round

**A's flaw.** §4 claims the violence tax "changes the incentive shape of every fight" and predicts the pacifist bot's "death distribution inverts the greedy bot's — mostly `deaths_dark`, a minority `deaths_combat`." But §4's ACT verb "costs a normal turn ... same as any action," and CLAUDE.md's turn structure is fixed: player acts → `monsters_act()` → FOV, every turn, regardless of verb. A's own numbers say "rats calm in 1-2 ACTs... ogres take 3-4" — that's 3-4 full turns standing adjacent to a live, hostile ogre, which gets a free `monsters_act()` swing on *every one* of those turns, same as if the player had just stood still and bumped it. Nothing in A's design stuns, flees, or otherwise interrupts the monster during regard-building. The pacifist path may therefore be *more* combat-exposed than greedy (more turns adjacent to danger, not fewer), the opposite of what the pacifist-band gate in Batch 4 predicts — and if the sim bot proves that out, `REGARD_MAX` and the mercy pitch both need rework post-hoc, not pre-verified as A's roadmap assumes.

**B's flaw.** §4's `GhostState{game, source: File(Vec<u8>) | Socket(TcpStream)}` claims local ghosts and live multiplayer are "architecturally identical" and that "no new 'other player' entity kind is needed anywhere in `game.rs`." That elides the actual hard part of lockstep netcode: a `File` ghost is pre-recorded, so "advance one logged input per real player turn" is safe — the log never runs out mid-turn. A `Socket` peer's bytes arrive on *their* wall-clock, not synchronized to the local player's turn count at all. B never specifies what happens when the local player wants to act but the peer's byte for this turn hasn't arrived (stall? desync? drop?) — that's a turn-barrier/ready-check protocol, real netcode complexity, and B's B10 test ("two `GhostState`s exchanging bytes over a fake in-memory transport... asserting converged `state_hash`") only proves convergence *given* inputs already arrived in the right order, not that the order problem is solved.

**What I'd steal.**
- From A: the pure `fn ending_id(&Game) -> u8` endings matrix (§4) — cheap textual variety (multiple flavored end-states keyed off run stats) beats my current single win/death pair for almost no bytes, without adding a mercy system I've otherwise cut.
- From B: the auto-capture-on-death idea (§4, "every death produces a ghost with zero extra player action") — I'd shrink it to one echo, not a ghost system: on retry-same-seed, dim-render the player's own death tile/glyph via the existing `scale()` path (already in my §3 screen-feel toolkit), a near-zero-byte staging of my memory ledger instead of a bare counter.

**Revised stance.** The mercy-vs-monsters_act() gap in A and the socket turn-barrier gap in B are both real systems risk hiding under "it's just data/relay." That's exactly the case for my thesis: presentation-only changes (screen-feel, sprites, audio, the death-tile echo I'm stealing) are the lowest-risk category of change in this codebase precisely *because* they don't touch turn structure or channel discipline — Batch 2's fixed finding shows the systems surface is already fragile enough without adding ACT/regard or live sockets on top. I'm not conceding ground; A and B's flaws are the same flaw wearing different clothes: both bet on new turn-affecting mechanics whose balance isn't provable until built.
