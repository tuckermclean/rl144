## Project shape

- Single binary crate, split along engine seams (keep this module count; don't fragment further without justification):
  - `src/rng.rs` — Rng, h64/channel, fnv (the frozen hash primitives)
  - `src/content.rs` — generic content helpers (theme_pick/theme_for/lore_line/ghost_label_idx) plus the engine's own UI-chrome palette; the authored data itself lives in the active cartridge (see `src/gamedef.rs`/`src/games/` below)
  - `src/gamedef.rs` — the `GameDef` data contract (batch 7): pure types, zero logic — `MonsterDef`/`ItemDef`/`ThemeDef`/`RoomKindDef`/`VaultDef`/`AuthoredFloorDef`/`BalanceDef`/`WinDef`/`StringsDef`/`GiveRule`/etc. Every game-specific fact the engine needs is a field on one of these.
  - `src/games/` — the cartridge(s). `src/games/mod.rs` is the one file allowed to name a specific game module; it re-exports `pub(crate) use contractor::GAME`. `src/games/contractor.rs` is cartridge #1: one `pub(crate) const GAME: GameDef` holding every monster/item/theme/vault/authored-floor/balance/win/string fact. See the Cartridge doctrine section below.
  - `src/game.rs` — engine core: tiles, entities, Game, worldgen, turns, `apply_input` (the client input-vocabulary boundary); reaches game data only via `crate::games::GAME`
  - `src/headless.rs` — dump/solve/sim/world_hash (verification tooling)
  - `src/save.rs` — save format, replay, state_hash (state-is-deltas layer)
  - `src/render.rs` — cell-grid presentation core: composes the 80×30 `Cell{ch,fg,bg}` grid (the part a DOS/mobile port swaps is a backend, not this file)
  - `src/backend_minifb.rs` — minifb window backend: rasterizes cells to the pixel framebuffer via font8x8, owns the window loop
  - `src/backend_term.rs` — ANSI terminal backend: hand-rolled termios raw mode, dirty-cell ANSI encoder, owns the terminal event loop
  - `src/main.rs` — arg parsing, backend dispatch, tests
- Two dependencies: `minifb` (window + pixel buffer) and `font8x8` (const glyph data), both `optional = true`, both pulled in only by the `backend-minifb` cargo feature. `font8x8` uses only its always-compiled `legacy` module (`BASIC_LEGACY`/`BOX_LEGACY`/`BLOCK_LEGACY`, plain `[u8;8]` tables with no `unicode` cfg gate) — glyphs are looked up by direct codepoint-offset indexing, not the `unicode` feature's `FontUnicode`/`UnicodeFonts` search API, so that feature is dropped from `Cargo.toml` entirely (bit-identical glyphs, smaller binary). Treat the dependency list as frozen. Adding a crate requires demonstrating (a) it can't be hand-rolled in <150 lines, and (b) the size cost, measured (see below).
- **No engines.** Bevy, ggez, macroquad, SDL bindings, etc. are all disqualified by the budget. No `serde`, no `rand` (we have xorshift), no `image` (we have no image files).
- **No asset files.** All content is procedural or `const` data in source. If you want art/audio, generate it at runtime or embed it as compact const tables. Adding a `assets/` directory is a design smell here.

## Core/crust doctrine

The core — `rng`/`content`/`gamedef`/`games`/`game`/`save`/`headless`/`render` — has **zero platform calls and zero `cfg` blocks** (the one exception, per the Cartridge doctrine below, is `games/mod.rs`'s cartridge-selection wiring, which is wiring, not engine logic). Its entire surface to a frontend is four things: `render_cells(&Game, Screen, &mut [Cell])` (the 80×30 `Cell{ ch: u16, fg: u32, bg: u32 }` grid), `scene(&Game) -> Vec<SceneEntity>` (batch 4: derived-only sprite-level presentation data — kind/x/y/facing/anim_phase/light_pct per visible entity, `SpriteKind` including `Echo` for the last-death marker — additive to `render_cells`, not a replacement; a flat character grid can't carry facing or animation continuity, which is exactly what a future sprite backend needs), `Game::apply_input(u8)` (the input-byte vocabulary, now 0–16 — see Code conventions below), and the headless entry points (`--dump`/`--solve`/`--sim`/`--replay`/`--render-frame`/etc). Backends (`backend_minifb.rs`, `backend_term.rs`) may only *consume* that surface — turn cells/scene entities into pixels or ANSI escapes, turn keystrokes into input bytes. A backend that needs a new core API is a real design decision, not a drive-by: justify it in the status entry.

**Presentation-only exclusion set — named doctrine, not a one-off list.** `Game::killer`, `Game::echo`, `Game::facing`, `Game::fx_hit`, and (batch 8 T1) `Game::mcguffin_last_line_turn` are NEVER hashed (`state_hash`), never dumped (`level_dump`), never saved (`save_bytes`'s input log doesn't need them — they're all derived or reconstructed, not player-chosen state). All five are documented together at `state_hash`'s doc comment in `save.rs`, which is the one place this set is enumerated. Anything added to this set in the future gets documented there too, in the same comment, not scattered across the field's own definition site.

`cfg` is allowed only inside backend modules and the dispatch/wiring code in `main.rs` (the `compile_error!` guards, `mod backend_minifb`/`mod backend_term` gating, and the call site that invokes whichever backend's `run(...)`). Exactly one backend cargo feature must be active at a time: `backend-minifb` (default) or `backend-term`, enforced by two `compile_error!`s in `main.rs` — one for zero backends, one for both. To build the terminal flavor: `cargo build --release --no-default-features --features backend-term --target-dir target/term` (a separate target dir so it never clobbers the default build's artifacts that `make size` measures).

Core hashed state stays explicit-width (`i32`/`u32`/`u64`) — no `usize` in anything feeding `state_hash`. This is 32-bit-port prep: `usize` in a hash means the hash changes width depending on target, which breaks replay portability. `usize` is fine for pure grid-index/array-length plumbing (`idx()`'s return type, loop counters); it must never land in a `Game` struct field that gets hashed.

## Cartridge doctrine

**The engine is generic; the game is data.** Landed batch 7 T1 (`c83606b`, "the unbraiding") as a pure, byte-identical refactor. `src/gamedef.rs` defines `GameDef` — a pure-data contract, zero logic — plus its component structs (`MonsterDef`, `ItemDef`, `ThemeDef`, `RoomKindDef`, `VaultDef`, `AuthoredFloorDef`, `BalanceDef`, `WinDef`, `StringsDef`, `GiveRule`, ...). `src/games/contractor.rs` is cartridge #1: ONE `pub(crate) const GAME: GameDef = GameDef { ... }` holding every monster/item stat, theme, vault, authored floor, balance constant, win-condition parameter, and player-facing message template. `src/games/mod.rs` is the one file in the crate allowed to name a specific game module; it re-exports `pub(crate) use contractor::GAME`, and every engine file (`game.rs`, `headless.rs`, `render.rs`, `save.rs`, `content.rs`) reaches game data only via `crate::games::GAME`, never by importing `contractor` directly.

- **Engine code must stay grep-clean of game nouns.** No occurrence of `rat`/`goblin`/`ogre`/`cheese`/`coat`/`towel`/`amulet`/`sword`/`potion`/`torch`/etc. as an identifier or *string literal* may appear in `rng.rs`, `game.rs`, `gamedef.rs`, `headless.rs`, `save.rs`, `render.rs`, `content.rs`, `backend_minifb.rs`, or `backend_term.rs`. This is a standing rule, checked by grep on every change to those files — not a one-time batch-7 cleanup. (Doc *comments* explaining a concept in plain English, e.g. "the torch burns down," are not the concern; a hardcoded string that would actually render or hash is.) `main.rs`'s test module is exempt — it's the crate's own test-and-wiring surface and legitimately reaches cartridge-specific named indices (`RAT`, `GOBLIN`, `CHEESE`, ...) for test fixtures.
- **A second game = a second `GameDef` file.** Add `src/games/<name>.rs` shaped like `contractor.rs` (one `pub(crate) const GAME: GameDef`), then flip `src/games/mod.rs`'s `pub(crate) use` (or gate it behind a compile-time selector — `cfg` is permitted in `games/mod.rs` exactly as it's permitted in `main.rs`'s backend wiring, since this module IS wiring, not engine logic). No second game ships yet; the shape exists so one obviously could.
- **GameDef does not enter world identity mechanically.** Channels are still keyed off the seed exactly as before the split; goldens freeze `GAME`'s data + the worldgen algorithm together, the same way they always froze hardcoded constants + algorithm. Swapping cartridges is therefore also a MAJOR-version event for any seed/save that crosses it, governed by the same doctrine as a worldgen change (see Seed compatibility below) — it just isn't a distinct mechanical hook today.
- **A leaked game-specific literal in engine code is a real bug, not a style nit** (see the batch-7 T3 status entry: a hardcoded `"Torch ["` in `render.rs`'s status-bar draw was exactly this — pulled into `StringsDef::resource_label`, set to `"Torch"` in `contractor.rs`). When you find one, move the string to the appropriate `GameDef` field (usually `StringsDef`) and have the engine read `GAME.<field>` instead.

## Size budget rules

Current baseline (Linux x86_64, Ubuntu rustc 1.75):
- stripped release build: ~506 KB
- after `upx --best --lzma`: ~166 KB

The number that counts against the 1.44MB limit is the **UPX-packed binary** (plus any asset files, of which there must be none).

After ANY change that adds code or data:

```sh
cargo build --release
ls -la target/release/rl144          # unpacked size
cp target/release/rl144 /tmp/p && upx --best --lzma -qq /tmp/p && ls -la /tmp/p
```

Report both numbers in your summary of work. Rules of thumb:
- A feature that adds >50 KB packed needs an explicit justification.
- If packed size ever exceeds 1 MB, stop adding features and shrink first (`cargo bloat --release` is the tool; it may need installing).
- Never remove the size flags in `Cargo.toml` (`opt-level = "z"`, `lto`, `codegen-units = 1`, `panic = "abort"`, `strip`). If a change requires unwinding panics or debug symbols, it's the wrong change.
- Do not commit UPX-packed binaries into `target/`; pack into a copy. Packed binaries in the repo root named `rl144-<platform>-upx` are release artifacts.

## Verification (headless-first)

There is no display in CI/container environments. The window cannot be opened there. Therefore:

- **`--dump` mode is the test harness.** `./target/release/rl144 --dump --seed N` generates all 5 depths of seed N and prints each as ASCII (theme header + lore line + map + `monsters=N items=M`). It must always work and must not open a window. Without `--seed` it uses a time-derived seed.
- **Dump legend** (`headless::level_dump`, drawn in this layering order — later entries win a shared tile): `#` wall, `.` floor, `>` down-stairs, `<` up-stairs (player `@` sits on top of it), `*` portal (batch 6 T1 — a door to another world; walking onto it only logs the destination, transit is waiting on it), `^` pit (batch 6 T2, sokoban — impassable, a pushed block filling it reverts the tile to `.`), `x` goal (batch 6 T2, sokoban — walkable, a locked block absorbs into it and it reverts to `.`), `!` potion, `)` sword, `&` amulet, `?` lore item, `r`/`g`/`O` rat/goblin/ogre, `B` push-block (batch 6 T2 — drawn after items, so a block can hide one), `@` player.
- **`--solve N`** (default 10000): the winnability + difficulty gate. Generates all depths per seed, BFS-checks the exit is reachable, computes the round-trip walk budget (descend ×1 + climb out ×2 per step), prints JSON stats (`min/p50/p90/p99/max`, worst seed), and exits nonzero on any unwinnable seed or drift outside `tests/solver-band.json`. Run it after ANY worldgen-adjacent change. With `--report` it prints stats and exits 0 without gating (the re-baselining flow).
- **`--sim N`** (default 5000) is now a GATE like `--solve`, band in `tests/sim-band.json`: `win_pct` in `[10,25]`, `deaths_dark` a raw count in `[2,40]` (re-baselined batch 4 after the violence tax, was `[1,500]` — a percent-share band would floor to 0 at this scale — dark deaths are ~0.1% of total deaths, see status log). The minority-vs-combat check (`deaths_dark < deaths_combat`) and `stuck == 0` are structural code checks, not tunable JSON. With `--report` it prints stats and exits 0 without gating, same re-baselining pattern as `--solve --report`. Band calibrated for 5000 seeds; a different `N` needs its own calibration pass, not a reuse of these bounds.
- **`--sim N --policy pacifist`** (batch 5, DECISION.md item 3's gate): the same greedy bot with one change — a routed step onto a non-calm monster emits a talk byte instead of a bump-attack. Gated against `tests/pacifist-band.json`, `win_pct` in `[5,20]` (re-baselined for the parley addendum; measured 7.9% = 394/5000). `make sim` runs both policies; the greedy band and its numbers must not move when the pacifist policy or its band changes — they share the engine, not the gate.
- **`--render-frame --seed N [--ascii]`** (backend-term builds only): pure cell-grid → ANSI-escape byte stream written to stdout, no TTY required (safe with stdout redirected to a file) — this is both the frame-golden capture command and the terminal encoder's verification surface. Frozen by `tests/golden/frame_seed_1.bin`, `frame_seed_42.bin`, `frame_seed_1_ascii.bin`.
- **`--daily`**: shared seed of the day (`h64(days_since_epoch, ["daily"])`). Precedence: `--seed` > `--daily` > launch entropy.
- **`--replay <file>`**: replays a save headlessly and prints an FNV state hash + summary JSON. Two replays of one save must hash identically — if not, channel discipline broke. **`--load <file>`** resumes a save in the window (replays headlessly first).
- **`make check`** runs the whole gate in order: build, test, test-term, goldens, frames, xhash, solve, sim, size. **`make targets`** prints the size scoreboard (stripped/packed/% of budget, per backend — each backend's packed size is checked against the same 1,474,560-byte budget independently). **`make xhash`** is the cross-backend determinism gate: replay the committed fixture `tests/fixtures/ref.sav` through both backend binaries and require an identical state hash — it compares the two backends to each other (not to a frozen constant), which is the actual proof that the core/crust seam is real and neither backend's plumbing perturbs replay.
- Any new system (new gen algorithm, new entity type, new item) should be observable via `--dump` or a new headless flag. Extend headless modes rather than claiming "it probably works."
- Sanity checks worth running after gen changes: `--solve 10000` green; every depth has one `<` at the entrance; depth 1–4 have exactly one `>`; depth 5 has the amulet and no `>`. (In dumps the player `@` sits on top of the entrance `<`.)
- Logic that can't be seen in a dump (combat math, FOV symmetry) should get `#[test]` functions. `cargo test` must stay green. Tests are free — they don't ship in the release binary.
- Window-path changes (input handling, frame pacing, minifb API usage) can only be compile-checked headlessly. Flag them explicitly in your summary as "needs interactive playtest" — do not report them as verified.

## Seed compatibility (MAJOR-version doctrine)

Worldgen output is a public API, frozen by the golden fixtures in `tests/golden/` (full 5-depth dumps for seeds 1, 2, 3, 42, 1337; `cargo test` string-compares them). Any change that diffs a golden **map layout** — the channel hash constants, the tag scheme, the order or count of draws on a worldgen/spawns/vault/theme channel, room/corridor/placement logic — breaks every seed and save in the wild. That is a MAJOR version bump requiring explicit human sign-off, never a drive-by. To re-baseline after sign-off: regenerate with `--dump --seed N > tests/golden/seed_N.txt`, re-run `--solve 10000`, re-commit `tests/solver-band.json`, and re-derive `START_LIGHT` from the new worst-case budget (its comment documents the derivation — keep it current). Dump-format-only diffs (headers, flavor lines) are not seed-breaking but still need the goldens regenerated; say so in the commit message.

## Code conventions

- **All randomness flows through named channels.** `channel(seed, &[tags])` (FNV-1a-64 + finalizer → xorshift64 `Rng`) isolates streams: `worldgen`/`spawns`/`vault`/`theme` are keyed per depth and frozen by the goldens; `combat`, `ai`, `flavor`, and `parley` (batch 5, the talk-receptivity roll) are per-run streams on `Game`. Never let one channel's draws leak into another — combat rolls must not perturb worldgen (there's a test). Never call system entropy mid-game; the only entropy is the launch-time seed.
- Coordinates are `i32`, grid indices via `idx(x, y)`, bounds via `in_map(x, y)`. Don't introduce a second coordinate convention.
- Entity/item stats live in the `Monster::stats`-style const match tables, not scattered magic numbers. New content = new table row.
- Turn structure: player acts (burning light via `spend_turn`) → `monsters_act()` → `compute_fov()`. Preserve this ordering; monsters must never act on a level the player just arrived on (see the early `return`s after stair transitions — they're intentional).
- Light is the run's clock: 1 per turn, 2 while carrying the amulet; 0 = death in the dark, and the lose check runs **before** the win check. `START_LIGHT` is solver-derived — its comment documents the derivation; don't retune it by feel.
- The win is a round trip: the amulet is picked up on depth 5 and carried to depth 1's `<`. Visited depths persist via `LevelState` snapshots — the climb out is through the world the player left. Don't regenerate visited levels.
- Bumping a wall costs no turn and burns no light (intentional). Pickup is on walk-over (intentional for v0; an inventory is a v1 discussion, not a drive-by refactor).
- Flavor is grounded: theme lore/adjectives may only restate things the engine did — never invent entities, exits, or events. Themes are const tables (`THEMES`); vaults are const strings (`VAULTS`) with the legend documented at the definition.
- Saves are seed + input log (`RL14` header format in `save_bytes`), never serialized world state. Anything that makes replay diverge from live play is a bug by definition.
- **The 80×30 cell grid is engine API; the window is presentation.** `COLS`/`MAP_H` are baked into `idx()` and worldgen — the grid must never follow the window size. Frontends scale the fixed 640×360 buffer (minifb: `resize` + `AspectRatioStretch`); a DOS/mobile port swaps the backend module, not the grid. Likewise the input-byte vocabulary is the platform boundary: any frontend that produces those bytes is a valid client. It is now **0–16** (batch 8 T1 added byte 16): 0–3 move/bump-attack, 4 wait, 5 restart (reroll to a new seed, `INPUT_RESTART`), 6 retry-same-seed (`INPUT_RETRY`, batch 4/save v2), 7–10 talk-N/S/W/E (batch 5, byte order mirrors the move bytes), 11–14 give-N/S/W/E (batch 7 T2, byte order mirrors move/talk), 15 use (batch 7 T2, self-apply the top of `Game.held`, no direction), 16 put-down (batch 8 T1, story §9-D, `Game::put_down` — self-apply like 15, no direction, sets the carried objective back into the current tile's item list and its carry-burn multiplier reverts to base automatically via `spend_turn`) — 5 and 6 are reconstruction bytes handled in `replay()`/the backend layer, not ordinary `apply_input` moves; 7–10, 11–14, and 16 are ordinary `apply_input` moves that route to `try_talk_player`/`try_give_player`/`put_down` instead of a bump-attack. A frontend turns a `t`(or `a`)+direction chord into one logged talk byte, `g`+direction into one logged give byte, a bare `u` into byte 15, and a bare `p` (no chord, batch 8 T1) into byte 16; the armed/disarmed chord state itself is frontend-local chrome, never engine state (same status as the F5 double-press save-overwrite confirm) — it's why the chord-reading helpers live in the backends, not `game.rs`. The title screen documents all of these ("Give: g+dir    Use: u    Put down: p") — a batch-7 T2 finding was that this legend line went in without the fix landing in the same commit; it's fixed now (`a249dd2`), and batch 8 T1 extended the same line with the put-down key rather than repeating the omission.
- Saves are `SAVE_VERSION 5` (`save.rs`, bumped from 4 in batch 8 T1 when put-down joined the vocabulary): `parse_save` accepts version bytes `1..=5` and replays each byte-identically (a v1 log can never contain byte 6, a v1/v2 log can never contain bytes 7–10, a v1/v2/v3 log can never contain bytes 11–15, a v1/v2/v3/v4 log can never contain byte 16, so accepting older versions changes nothing about their replay). The version bump's own point is the opposite direction: an *old* binary handed a newer-version save must reject it cleanly rather than silently ignoring bytes it doesn't know and diverging from the log that produced them. Don't drop old-version acceptance, and don't skip a version bump when the vocabulary grows again.
- **Held items are a LIFO list, not an inventory UI** (`Game.held: Vec<u8>`, batch 7 T2, story §9-A — "a small held-items list, hashed, no grid UI"). `ItemDef::on_pickup` is `Hold` (pushed onto `held` on walk-over) or `Consume` (the original v0/v1 instant-apply-and-vanish behavior, still used by the sword/amulet/lore items) — a per-item authored choice, not a global switch. GIVE (bytes 11–14, `Game::try_give_player`) and USE (byte 15, `Game::use_item`) both act on `held.last()` only: GIVE consults `GameDef::give_table` for a `(item, target monster kind)` row (regard delta, optional heal-to-full, optional line, whether the give consumes the item) and no-ops gracefully with a `StringsDef` feedback line if there's no monster in the offered direction, nothing held, or no table row for that pair; USE applies `ItemDef::on_use` (`Heal`/`Light`, batch 7's only two variants) to the player and no-ops with its own feedback line if nothing's held or the held item has no self-use. Both are ordinary turns (`spend_turn`, no violence tax — offering or using an item isn't violence) and both pop `held` on a landed effect. `held`'s length and contents are hashed (`state_hash`) — order matters for what GIVE/USE will act on next, so it's run-defining state, not presentation. This batch moved the **potion** from walk-over `Consume` (instant heal) to `Hold`+USE — the greedy sim bot was updated to emit USE when hurt and holding a potion (a real bug was caught and fixed here: an earlier bot check used `held.contains()` instead of `held.last()`, which could fire USE on a potion buried under another held item that GIVE/USE can't actually reach — fixed before the sim numbers below were taken as the baseline).
- **The McGuffin talks through `CarryEvent` dispatch** (batch 8 T1, story §9-B/C/D — "voice two comes online"). `GameDef::carried_lines: &[(CarryEvent, &[&str])]` is a generic engine-side hook — an event tag to line-pool table — plus `GameDef::carried_preamble: &[&str]`, a fixed intro sequence. `Game::carry_event(ev)` fires ONLY while the objective is being carried; if `carried_lines` has no row for `ev`, or that row's pool is empty, it's a provable no-op (no RNG draw, no mutation) — the invariant that kept T1 byte-identical to pre-batch-8 with an empty-pool cartridge, and the same invariant that lets a future cartridge ship zero McGuffin content for free. Line selection draws from `flavor_rng` only, never `combat`/`ai`/`parley`/worldgen channels. **§9-C, the pickup register**: at first pickup, `CarryEvent::PickedUpBloody` fires if `kills > spared`, else `PickedUpMerciful` — preceded, once, by `carried_preamble` printed in fixed order (never shuffled, never repeated on a later pickup). A re-pickup after a `put_down` fires `CarryEvent::PickedBackUp` instead. **The climb ladder**: `CarryEvent::StairsUp` is dispatched on every carried ascent, indexed by `Game::speech_attempts` (a hashed `u8`) rather than drawn at random — the index is read BEFORE the counter increments, so the first carried ascent reads rung 0, matching an ordered "one true sequence" pool rather than a shuffleable one. **New hashed state**: `speech_attempts` and `objective_dropped` (both run-defining — they change which `CarryEvent` a future dispatch fires — so both join `state_hash`, same status as `held`/`regard`/`calm`/`spared`); `Game::mcguffin_last_line_turn` (the rate-limit clock, at most one line per `carried_lines`' configured turn gap except the always-speak events `PickedUpBloody`/`PickedUpMerciful`/`PutDown`) is presentation-only and lives in the `killer`/`echo`/`facing`/`fx_hit` exclusion set instead (see that doctrine above). **Put-down** (byte 16, `Game::put_down`, story §9-D) is legal anywhere the objective is carried: it re-enters the objective into the current tile's item list, flips `objective_dropped`, fires `CarryEvent::PutDown`, and the carry-burn light multiplier reverts to base automatically through the existing `spend_turn` path (no separate revert logic needed — carry-burn was already conditional on `has_objective`). A found-during-review gap: the brief had assumed §9-C was "just T2 data," but the pickup-register branch needed an engine hook of its own (the kills-vs-spared read at first-pickup time) — folded into T1 as a fix round rather than punted to T2, since T2 owns only cartridge data, not engine logic.
- **Ghost files** (`rl144-<worldhash>.ghost`, `RLG1` magic) are the death-capture sibling of a `.sav`: written by pure builders (`ghost_bytes`/`parse_ghost`) in `save.rs`, zero I/O in that module — the backends own the actual file write, same core/crust split as everything else. Auto-written on death when the End screen's R/N/Q is pressed (never on a win); latest death overwrites, one ghost per world. Labels come only from the const `GHOST_LABELS` table via `content::ghost_label_idx` (deterministic from outcome/depth, no new RNG channel, no free text ever — same discipline as flavor/lore). Phase 1 substrate only: no ghost rendering or playback yet, so `Ghost`/`parse_ghost` are `#[allow(dead_code)]` pending that later task.
- **Violence tax** (`VIOLENCE_TAX` in `game.rs`, batch 4, DECISION.md item 1 — the mercy economy's first brick): every bump-attack burns 1 extra light on top of the normal per-turn cost, folded into `spend_turn`'s single deduction so the light-0 death check still runs exactly once and lose-before-win ordering holds — a killing blow whose tax lands light at or below 0 is correctly a darkness death. Wall bumps still cost no turn and burn no light. The tax stands unchanged through batch 5 — mercy is priced on top of it, not instead of it.
- **Mercy is a verb and the verb is talk** (batch 5, DECISION.md item 3, the Henson ruling — "if you could talk to a rat, you could give a rat mercy"). A talk byte (7–10, directional) at a monster rolls its receptivity — the full formula with per-term provenance lives at `receptivity()` in `game.rs`; reference it there, don't duplicate the math in comments or docs. A landed roll: `+1 regard` and the monster is stayed this turn (it's listening, so it doesn't get to swing — implemented as a transient per-turn parameter threaded through `monsters_act`, never stored, never hashed, cleared every turn; other monsters act normally, so crowds stay dangerous). A failed roll: no regard, and the monster is *not* stayed — it acts normally, meaning it can retaliate the same turn. Talking always costs the turn either way, tax-free; talking carries risk precisely because failure buys nothing. A monster becalms (`Monster.calm`, `Game::spared` increments) at `Monster::talk_threshold(kind)` — rat 2, goblin 3, ogre 4, unchanged since batch 5 T1. A becalmed monster never attacks or chases and yields (position-swaps, no damage, no tax) on a bump instead of being attacked. `regard`, `calm`, and `spared` are all hashed state (`state_hash`) — mercy is run-defining, not presentational; the existing `killer`/`echo`/`facing`/`fx_hit` exclusion set is unchanged. Talk rolls come from the `parley` channel (`channel(seed, &["parley"])`), a new per-run stream on `Game` alongside `combat`/`ai`/`flavor` — never `combat_rng`, never worldgen.
- **Portals are doors to other worlds** (batch 6 T1, `Tile::Portal`, dump glyph `*`). Placed via the worldgen channel, ~1/4 per depth, never in the entrance or exit room. A portal's destination is rolled ONCE at generation time and cached (`Game::portal`/`LevelState::portal`), never re-derived: either `Dest::World(seed, whash)` — `seed = h64(world_seed, ["portal", depth_tag, index_tag])`, `whash` memoized via `headless::world_hash(seed)` — or `Dest::Floor(u8)` indexing `content::AUTHORED_FLOORS` (~1/3 of rolls when any floors exist). The whole multiverse is a pure function of the root seed, frozen by goldens like everything else. Walking ONTO a portal never transits — it only logs the destination (theme label + world hash, or the authored name), grounded because the engine proves it by generating the destination. Transit is pressing wait (byte 4) while STANDING on the portal — a deliberate reuse of the existing wait byte, so the input vocabulary doesn't grow and neither sim bot (which never emits wait) can transit by accident. Light, hp/maxhp/atk, kills, spared, and `has_amulet` are GLOBAL across worlds — the torch is the multiverse's one clock. The amulet exists only in the ROOT world's depth 5, and the win check fires only on the ROOT world's depth-1 `<`. In any non-root world, depth-1's `<` (or a `Floor`'s sole `<`) is the return portal: walk-on, like stairs, lands you back on the source world's portal tile. `Game::worlds` (a `WorldId -> WorldState` map, insertion-ordered by first-LEFT) holds every visited non-root world's saved `LevelState` stack plus its own provenance (`from`), so a multi-hop portal chain unwinds correctly; all of it is hashed in `state_hash`. Solver/sims are structurally unaffected: root-world BFS reachability never routes through a portal (transit needs an explicit wait), so `--solve` and both `--sim` bots ignore portals entirely — a portal-diving bot policy is future work, not this batch's.
- **Authored floors** (batch 6 T1, `content::AUTHORED_FLOORS`) are hand-built, singular, non-random single-level maps a portal's `Dest::Floor` can point to — no depths, no stairs, no amulet, just a `<` that returns to the source portal. Same floor reached from two different worlds is the SAME floor (keyed by `WorldId::Floor(index)`, not by which portal led there), so its visited state (kills, pickups) persists across every door onto it. This is the enabling tech for a future fixed-overworld screen (story `docs/story/STORY-COMPILE-v1.md` §9-J) — don't re-derive an equivalent mechanism elsewhere.
- **Sokoban** (batch 6 T2, ported in spirit from `golem/topdown-puzzle`'s `shared/push.js`, vault-only): pushing a block (`Game::blocks`, walk-into) costs a turn of light like any move; a push chain is capped at `MAX_PUSH_CHAIN = 2` (a 3rd consecutive block refuses the whole push, no turn spent). A chain's farthest member landing in a `Tile::Pit` is destroyed and the pit FILLS (`Tile::Floor`) — the sokoban bridge; landing on a `Tile::Goal` LOCKS it there instead (absorbed out of `Game::blocks`, tile reverts to `Tile::Floor`) per `docs/story/STORY-COMPILE-v1.md` §9-H ("a block pushed onto a goal locks and opens/reveals the room's reward"). Blocks are hashed state (position matters) and persist per seed — "someone was here, someone was bad at this." A pit is impassable to player and monster alike (refused like a wall, no turn); a goal tile is ordinary floor for every game-logic purpose except the push-lock and its distinct render color. Abandoning a sokoban room mid-solve is always legal — these rooms gate loot only, never the level exit (`Game::gen_level` excludes a sokoban vault's own center from exit-room selection, and the solver proves the exit stays reachable on every seed regardless).

## Subagent dispatch discipline

This project is built almost entirely through dispatched subagents (one implementer per task → a reviewer → fix round(s) as needed → next task; a final whole-branch reviewer before every merge). The discipline is about model tier and parallelism, not about avoiding blocking calls:

- **Model tiers**: Fable or Opus for planning and final review (writing/adjudicating batch briefs, the whole-branch review before a merge — judgment-heavy, low-volume work). Sonnet for implementer/worker subagents — the actual per-task code-writing, especially anything parallelizable (independent tasks with no shared state, e.g. two content-drafting tasks, or a code task alongside a docs task). Dispatch parallel Sonnet workers whenever tasks don't depend on each other; don't serialize work that doesn't need it.
- **Running out of session/usage limit mid-batch is expected and fine, not a failure mode to engineer around.** It is recoverable: use `ScheduleWakeup` (or the `/loop` skill in self-paced/dynamic mode) to schedule a check-in — every 20–30 minutes is a reasonable default cadence — so work resumes automatically once the limit clears, picking back up from `git status`/`git log` and the local `.superpowers/sdd/progress.md` ledger. Don't contort dispatch prompts trying to prevent hitting the limit; just make sure state (branch, commits, ledger) is always in a resumable place when you stop.
- **State the acceptance numbers up front when they're knowable** (a pure refactor's expected byte-identical gate values; an unchanged sim/solve baseline a task must not perturb) so the implementer can self-verify against a concrete target instead of an implicit "should still work." Independently re-run the empirical claims in every review rather than trusting a report's numbers — this repo's review passes have caught real bugs this way (a bot stuck in an infinite loop, a killing-blow visual effect that silently never rendered, a save-format claim that was actually correct but needed tracing to confirm).

One session-management habit worth keeping: when a batch's story/design work implies a *new* mechanism (not just new content), write it into the relevant staging doc with an explicit "needs its own sign-off" flag rather than quietly building it — see how `SPACES-DRAFT-v0.md`'s screen-link glyph is marked above. Undersized asks get missed; flagged ones get their own brief.
